# 한글 HWPX 손상 판정 연구 보고서

## 개요

이 문서는 한글(Hancom Hangul)이 특정 HWPX 파일을 "손상되었을 가능성이 있다"라고 판정하며 열기를 차단하는 현상을 재현하고, 그 원인을 실험적으로 좁혀 나간 결과를 정리한 보고서다.

목표는 단순히 경고를 피하는 것이 아니라, 어떤 수정이 손상 판정을 유발하는지 제어 가능한 규칙으로 바꾸는 것이다.

## 문제 정의

관찰된 현상은 다음과 같다.

- ZIP 구조상 정상인 HWPX도 한글에서 손상 또는 보안 경고로 차단될 수 있다.
- 패키지 검증, XML 파싱, 재오픈 검증이 모두 통과해도 실제 한글은 열기를 거부할 수 있다.
- 따라서 "ZIP/XML이 유효하다"와 "한글이 신뢰하고 연다"는 서로 다른 판정이다.

이 문제는 특히 학교 양식처럼 복잡한 `section0.xml`을 가진 템플릿에서, 본문을 대량 수정할 때 재현되었다.

## 연구 가설

초기에는 다음 후보를 모두 열어 두고 접근했다.

1. ZIP 재압축 방식 차이
2. 패키지 엔트리 순서 또는 압축 플래그 차이
3. 일부 XML 재직렬화 과정에서 생기는 구조 변화
4. `hp:linesegarray` 같은 레이아웃 캐시 불일치
5. 표, 그림, 컨테이너 같은 복합 객체를 품은 문단의 오염
6. 해시 또는 파일 전역 무결성 검사

중요한 점은, 특정 원인을 미리 정답처럼 가정하지 않고 각 조건을 분리해서 실험했다는 것이다.

## 핵심 실험 결과

### 1. 전역 해시 기반 차단 가설은 주원인으로 보이지 않았다

`hancom_damage_forensics.py`는 ZIP 패킹 방식만 바꾼 파일, 미리보기만 바꾼 파일, `content.hpf`만 바꾼 파일, `settings.xml`만 바꾼 파일, `section` 텍스트를 바꾼 파일을 분리해서 만들도록 되어 있다.

이 실험 구조의 의미는 다음과 같다.

- 만약 전역 해시나 파일 전체 서명이 절대 기준이면, 어떤 부분만 바꿔도 거의 같은 방식으로 차단되어야 한다.
- 하지만 실제로는 수정 위치와 수정 방식에 따라 결과가 달랐다.
- 즉 손상 판정은 "무조건 내용이 바뀌었는가"보다는 "한글이 보기엔 비정상적인 조합이 생겼는가"에 더 가깝다.

이 결과만으로 해시 검사를 100% 배제할 수는 없지만, 현재까지의 주원인은 아니다.

### 2. `linesegarray` 불일치는 독립적인 손상 유발 요인이다

문단 텍스트를 바꾼 뒤 해당 문단의 `hp:linesegarray`를 그대로 두면, 일부 문단에서 한글이 열기를 거부했다.

이후 같은 문단에 대해 수정된 문단의 `linesegarray`만 제거하면 정상적으로 열리는 경우가 반복적으로 확인됐다.

이전 실험에서 `linesegarray` 제거로 정상화된 단일 문단 인덱스는 다음과 같다.

- `26`
- `31`
- `40`
- `44`
- `65`
- `66`
- `77`

이 결과는 `linesegarray`가 단순 장식 정보가 아니라, 한글이 내부적으로 신뢰하는 조판 캐시이며 실제 텍스트와 어긋나면 손상 신호로 작동할 수 있음을 시사한다.

### 3. 복합 객체를 품은 문단은 일반 텍스트 치환 규칙으로 다루면 안 된다

보고서 실험에서 결정적인 실패 지점은 복합 문단이었다.

`hancom_report_forensics.py`는 보고서 매핑 대상 중 복합 문단을 다음 XPath로 식별한다.

- `./hp:run/hp:tbl`
- `./hp:run/hp:pic`
- `./hp:run/hp:container`
- `./hp:run/hp:ole`

실험상 복합 문단 인덱스는 다음 둘이었다.

- `48`
- `57`

특히 `57`은 표를 품은 상위 문단으로, 일반적인 `.//hp:t` 재귀 치환을 쓰면 표 내부 텍스트나 캡션까지 함께 망가질 수 있었다. 이 경우 `linesegarray`를 제거해도 실패가 계속됐다.

즉 복합 문단 문제는 레이아웃 캐시 불일치보다 더 근본적인 구조 오염 문제다.

### 4. 보고서 전체 변형 실험에서 성공 조건이 분명히 갈렸다

이전 실험 결과는 다음과 같이 정리됐다.

- `report_keep_lineseg`: 실패
- `report_drop_lineseg`: 실패
- `report_drop_lineseg_skip_complex`: 성공

이 결과의 의미는 명확하다.

- `linesegarray` 문제만 해결해도 충분하지 않다.
- 복합 문단을 일반 치환에서 제외해야 한다.
- 일반 문단에는 `linesegarray` 제거가 유효하다.
- 따라서 실제 제어 규칙은 최소 두 단계다.

## 원인 분석

현재까지의 가장 강한 설명은 다음 조합이다.

1. 한글은 HWPX를 단순 ZIP/XML로만 보지 않는다.
2. 문단 텍스트, 레이아웃 캐시, 복합 객체 구조, 패키지 메타데이터의 정합성을 함께 본다.
3. 특히 대량 수정 시에는 "구문적으로는 유효하지만 한글 내부 모델로는 비정상적인 상태"가 쉽게 만들어진다.

이 설명은 다음 관찰을 함께 설명한다.

- 패키지 검증은 통과하지만 한글 열기는 실패한다.
- 일부 문단은 `linesegarray` 제거만으로 복구된다.
- 일부 문단은 구조적 이유로 그 방법만으로는 복구되지 않는다.
- 단순 ZIP 재저장만으로 항상 실패하지는 않는다.

## 제어 규칙

실험을 통해 현재 제어 가능한 규칙은 다음으로 정리됐다.

### 수정 규칙

- 문단 텍스트 수정은 재귀 `.//hp:t`가 아니라 직접 `./hp:run/hp:t`만 수정한다.
- 표, 그림, 컨테이너, OLE를 품은 상위 문단에는 일반 텍스트 치환을 적용하지 않는다.
- 수정한 일반 문단의 `hp:linesegarray`는 제거한다.
- 복합 문단 내용이 꼭 필요하면 해당 객체 내부의 안전한 하위 노드를 별도로 지정해서 수정한다.

### 저장 규칙

- 가능하면 원본 ZIP 엔트리와 압축 메타데이터를 최대한 보존하는 방식으로 저장한다.
- 원본과 동일한 바이트를 유지할 수 있는 엔트리는 그대로 보존한다.
- 변경된 파트만 최소 범위로 다시 기록한다.

### 템플릿 규칙

- 대량 수정은 자유 본문 전체 치환보다 placeholder 기반 치환이 안전하다.
- 특정 템플릿에는 템플릿 전용 안전 문단 목록과 금지 문단 목록이 필요하다.

## 코드에 반영된 내용

[`scripts/create_ai_singularity_report.py`](D:\GitHub\jakal-docs\scripts\create_ai_singularity_report.py)에는 위 규칙이 실제 반영돼 있다.

주요 반영 내용은 다음과 같다.

- `set_paragraph_text()`가 `./hp:run/hp:t`만 직접 수정
- `remove_linesegarray()`로 수정 문단의 캐시 제거
- `PARAGRAPH_MERGE_MAP`으로 복합 문단 `48`, `57`의 텍스트를 각각 `47`, `56`에 병합
- `PARAGRAPH_REMOVE_INDICES`로 복합 문단 제거
- `save_document_compat()`로 원본 ZIP 메타데이터 최대 보존

이 방식으로 생성한 최종 보고서는 실제 한글 열기 검증에서 `opened` 상태를 확인했다.

## 한계

아직 확정하지 못한 부분도 있다.

- 한글의 정확한 내부 손상 판정 알고리즘은 비공개다.
- 일부 ZIP 메타데이터 차이가 판정 민감도에 미치는 영향은 추가 실험이 필요하다.
- 복합 객체 내부를 안전하게 자동 편집하는 일반 규칙은 아직 없다.
- 템플릿마다 위험 문단 위치가 다를 수 있어, 현재 규칙은 템플릿 독립적 해법이 아니라 강한 경험 규칙에 가깝다.

## 다음 단계

우선순위가 높은 다음 실험은 아래와 같다.

1. 동일 텍스트, 동일 XML, 상이한 ZIP 메타데이터만 가진 파일군 비교
2. 복합 문단 내부에서 안전한 하위 노드만 수정하는 실험
3. placeholder 기반 템플릿 편집 파이프라인 설계
4. 마지막 저장을 한글 COM `SaveAs`로 넘기는 하이브리드 파이프라인 비교
5. 템플릿별 위험 문단 자동 탐지 규칙 일반화

## 결론

현재까지의 결론은 다음과 같다.

- 한글 손상 판정의 주원인은 "파일이 바뀌었다" 자체가 아니다.
- 더 유력한 원인은 수정 후 남은 비정상적인 내부 정합성이다.
- 실험상 가장 중요한 두 축은 `linesegarray` 불일치와 복합 객체 앵커 문단 오염이다.
- 따라서 손상 현상은 이미 일정 부분 제어 가능하다.

즉, 문제는 랜덤한 호환성 이슈가 아니라 재현 가능한 문서 변형 규칙의 문제이며, 현재 수준에서도 안전 규칙을 적용하면 한글이 열 수 있는 형태로 생성할 수 있다.
